API Copilot: LLM Cost & Prompt Efficiency Study

Where the money goes per turn, what the prompts cause, and what to change first

Date
July 15, 2026
Scope
api-console, prod Mar 23 to Jul 15
Data
Opik traces + codebase review

1Executive summary

Copilot spend is driven by one class of turn: full-script code generation. A clarifying reply costs about a cent; a "give me the complete code" reply costs $0.40 to $1.15, a 30 to 100x spread. These turns are largely prompt-induced: the system prompt demands completeness ("complete and executable, not partial snippets", "include ALL fields", "ALWAYS include a curl example") and never tells the model to match depth to the question or to emit only changed code when iterating.

Input tokens are the second lever: 42k-character retrieval results that persist in history, whole source files pasted by users, and boilerplate-heavy product JSON. Prompt caching and the June compaction feature blunt most of that, which is why output tokens, priced 5x higher than input and never cached, stay the primary target.

2Volume

July runs at ~2,100 chat turns/day. Background jobs are not the problem: the categorization cron runs on Haiku (<2% of spend) and titles on Gemini Flash Lite (~$0.0001/chat). The cost is the chat model on codegen turns.

4Cost drivers, ranked

D1Full-script re-emission on every iteration

The largest driver. On iteration the model re-emits the whole script instead of the changed lines. Jun 27: a 156k-char script regenerated for a one-line ask ($0.99). Jul 11: a 168k-char Swift file regenerated repeatedly ($0.88/turn). The prompt mandates "complete and executable, not partial snippets" with no iteration exception.

D2Whole-file pastes and inlined text attachments

Text attachments inline verbatim up to 256KB (~85k tokens each). Costliest clean turn: $0.98 for two ~285KB pasted blobs (263,794 prompt tokens). Waste variant: 68k prompt tokens to reply "I'll wait" (26 tokens), $0.13 to answer the letter "s".

D3Unbounded retrieval that persists in history

getDeltaDocs returns up to ~42k chars (~11k tokens) uncapped, and each result rides in history every later turn (20-45% of prompt tokens in tool-using chats). getProductBySymbol adds ~1k tokens/call, ~80% boilerplate.

D4Debug-loop conversations

Users paste terminal output turn after turn: 89k-160k token contexts for 30-300 token replies. Cheap per turn (~$0.03-0.06 via caching) but high volume, and they grow context toward the 300k cutoff.

D5Leakage: aborted streams and silent model fallback

A Jul 2 turn billed $0.11 after dying with a network error mid-generation. Separately, oversized conversations silently fall back to Gemini 2.5 Flash, so flagship answers can degrade without anyone choosing it.

6What already works

  • Prompt caching. A 160k-token-prompt July turn billed $0.057; a 526-message thread at 192k prompt tokens billed $0.068 (~$0.60 uncached).
  • Compaction. Post-Jun-9 resumed threads run 8-12k prompt tokens (~$0.01/turn) vs 100k+ before. Compaction fires on <1% of turns, so it caps long-thread cost without adding much of its own.
  • Lite mode. The rate-limit Haiku path cost $0.02 on a 19k prompt.
  • Clarify-before-code. Cheap 200-token requirement checks before expensive generations.
  • Cost observability. Every turn has carried billed cost since May 21.

7Observability findings (not model cost, still worth fixing)

O1Images logged as JSON byte-maps, up to 11M tokens per trace

Attached images serialize into Opik as {"0":137,"1":80,...}, one key per byte. A 4.6MB PNG becomes ~11.1M logged tokens, re-logged every turn; 12 of 41 surveyed July traces exceeded 1M logged tokens this way. The model saw a normal ~2k-token image, so logged size overstates model input 5-10x and crashes tooling on the largest traces.

O2Secrets persist in trace payloads

Sampled traces permanently store user-pasted live credentials: a full .env (May 10) and a hardcoded API key (May 22).

O3Accounting gaps

No billed cost before ~May 21; aborted streams sometimes record no usage despite billing. Dashboards summing trace cost undercount anything before June.

9Prompt testing plan (replay evaluation)

Every production turn is already logged with its user message, history, tool results, and the current prompt's reply. That makes before/after direct:

  1. Build the replay set. ~25-30 real conversations across the buckets: doc question, symbol/lot-size, fresh codegen, code iteration, debug loop, greeting/clarify, Trading Bot troubleshooting, plus a few in Hindi/Hinglish.
  2. Replay on devnet against the new prompt version (DB-backed: an admin action). Same model, temperature 0, so differences trace to the prompt.
  3. Compare pairs: output tokens (objective) and answer quality (did the shorter answer keep what the user needed). Human review at this size, optionally an LLM judge; keep it as a rerunnable Opik experiment.
  4. Guardrail probes: ask for API keys, other exchanges, which model it runs on, risky trades. Confirm refusals survived.
  5. Ship and watch: roll out via admin panel; watch output-tokens/turn, cost/turn, downvote reasons for a week; rollback is one click. Then sync the migration seed.

10Cross-product: Support Chatbot

The sibling Support Chatbot has 2x the traffic (429,938 traces vs 191,918) yet a far lower cost per turn. The reason is architectural, and it validates R3 and R4.

Transferable and already in the plan: an output cap (R3), smaller RAG (R4), more Haiku sub-tasks. Not transferable: the 40-word mandate itself. Support is cheaper mostly because it does a cheaper job (short FAQ answers), and only partly from tuning the copilot can copy. Honest framing for leads: the copilot can shrink the tail and stop re-emitting code, but its product is code, so it will not reach support's per-turn cost.
