API Copilot: LLM Cost & Prompt Efficiency Study

Where the money goes per turn, what the prompts cause, and what to change first

1Executive summary

The copilot's LLM spend is dominated by a small class of expensive turns: full-script code generation. A typical clarifying reply costs about a cent; a "give me the complete code" reply costs $0.40 to $1.15, a 30 to 100x spread. These expensive turns are largely prompt-induced: the system prompt mandates completeness ("complete and executable, not partial snippets", "include ALL fields from the schema", "ALWAYS include a curl example") and contains no guidance to match response depth to the question or to emit only changed code when iterating.

On the input side, three compounding leaks inflate prompt tokens: retrieval results of up to 42k characters per getDeltaDocs call that then ride along in history on every later turn, users pasting whole source files and 256KB text attachments inlined verbatim, and boilerplate-heavy product JSON. Prompt caching and the June compaction feature mitigate much of this, which is why output tokens, priced at 5x input and never cached, remain the primary lever.

2Methodology & caveats

Six parallel investigations read the Opik project "API Console Prod" end to end: four sampled 15 chat turns each from late March-April, May, June, and July; one measured tool-call output sizes across ~20 tool spans; one dissected the largest logged traces. Every cost figure is OpenRouter's billed cost as recorded per turn (cache discounts included), not a token-price reconstruction. Codebase claims reference the repo at commit e3eb292.

Caveats that bound the numbers

  • No data before March 23. The Opik project was created then; January and February do not exist in any project in the workspace.
  • Cost telemetry begins ~May 21. Earlier traces carry token usage but no billed cost. Spend before that must be reconstructed from tokens x price, and the model was different (see §3), so month-over-month cost comparisons are only meaningful from June onward.
  • Sampling is stratified, not random. Agents deliberately sampled both quick and slow turns, which overweights codegen. Class medians are robust; the class mix (and therefore total-spend estimates) carries wide error bars and is presented as scenarios in §5.
  • Trace counts are not comparable across eras. Before mid-May, tool calls and title generation were logged as separate root traces (~40% of rows); after, they nest inside the turn trace.
  • A handful of giant traces could not be fully read (the observability tooling itself fails on them; see §9). Their usage is inferred, and marked as such.

5Turn economics

Costs are sharply bimodal. Across the 44 Sonnet-era turns with billed cost, short turns (confirmations, clarifying questions, greetings) cluster at fractions of a cent to a few cents, while full-script codegen turns cluster between $0.24 and $1.15. Mid-range turns are comparatively rare.

6Cost drivers, ranked

D1Full-script re-emission on every iteration (output side, largest)

When a user iterates on code, the model re-emits the entire script. Evidence: a Jun 27 turn produced a 156k-char script for the ask "I need logic in this code don't delete my old code" ($0.99); a Jul 11 conversation regenerated a 168k-char Swift file multiple times ($0.88/turn); an Apr 30 turn streamed for 10.6 minutes emitting 52k completion tokens. The prompt requires this: "complete and executable (not partial snippets)", with no changed-sections-only rule for iterations. Six of 15 June samples had 22k-51k completion tokens.

D2Whole-file pastes and inlined text attachments (input side)

Text attachments inline verbatim up to 256KB (~85k tokens each). The single most expensive clean measurement: $0.98 for one turn carrying two ~285KB text blobs (263,794 prompt tokens, largely uncached). One May user pasted an entire codebase including a live .env; turns of 100k-545k prompt tokens recur in every month sampled. Related waste: "expensive no-ops", e.g. 68k prompt tokens to reply "I'll wait" (26 tokens), $0.13 to answer a message consisting of the letter "s".

D3Unbounded retrieval results that persist in history (input side)

getDeltaDocs returns top-5 merged chunks with no size cap: typically 5-25k chars, up to ~42k chars (~11k tokens) for order/bracket/option-chain topics, and the same giant chunks recur across many users' chats (bracket-order queries returned 18.6k/27.4k/41.9k chars in three different chats the same day). Once injected, results ride in history every turn: tool results were 20-45% of prompt tokens in sampled tool-using chats, and each tool round-trip re-sends the whole history (one 10-ticker turn billed 57.7k prompt tokens on a 23.4k-token context). getProductBySymbol adds ~1k tokens/call of which ~80% is boilerplate (zeroed fees, duplicated asset objects).

D4Debug-loop conversations (volume x input)

Users operate bots on VPSes and paste terminal output turn after turn; each turn re-sends an 89k-160k token context for a 30-300 token reply. Caching keeps these at $0.03-0.06 per turn, but they are numerous and grow context toward the 300k cutoff.

D5Leakage: aborted streams and silent model fallback

A Jul 2 turn died with a network error after 7.2 minutes and 5 doc lookups yet still billed $0.11: aborted long generations cost real money. Separately, the May 31 window-overflow turn was silently served by Gemini 2.5 Flash (573k tokens, $0.23), meaning the product's flagship answers can degrade without anyone choosing that.

8What already works

  • Prompt caching. Breakpoints on the system prompt and the last two user turns. A 160k-token-prompt July turn billed only $0.057; a 526-message thread at 192k prompt tokens billed $0.068 (roughly $0.60 uncached).
  • Compaction / Continue session. Post-Jun 9 resumed threads run 8-12k prompt tokens (~$0.01/turn) versus 100k+ in comparable pre-compaction threads.
  • Lite mode. The rate-limit Haiku path cost $0.02 on a 19k prompt; a meaningful lever if ever extended.
  • Clarify-before-code behavior. The prompt's confirm-intent rule produces cheap 200-token requirement checks before expensive generations.
  • Cost observability. Since May 21, every turn carries OpenRouter's billed cost; per-turn attribution made this study possible.

9Observability findings (not model cost, still costly)

O1Images are logged as JSON byte-maps, inflating traces up to 11M tokens

Attached images serialize into Opik as {"0":137,"1":80,...}, one JSON key per byte (~9 chars/byte). A 4.6MB PNG becomes ~11.1M logged tokens, re-logged on every turn of its thread; 12 of 41 surveyed July traces exceeded 1M logged tokens this way. The model itself saw a normal ~2k-token image, so logged size overstates model input 5-10x on image turns. Side effects: Opik storage bloat and traces the tooling cannot read at all (reads crash on 5M+ spans).

O2Secrets persist in trace payloads

Sampled traces permanently store user-pasted live credentials: a full .env (May 10) and a hardcoded API key (May 22). The assistant declined to use them, but Opik retains the raw text.

O3Accounting gaps

No billed-cost field before ~May 21; pre-mid-May usage keys differ (camelCase); aborted streams sometimes record no usage at all despite billing. Dashboards summing trace costs undercount anything before June.

11Prompt testing plan (replay evaluation)

Every production turn is already logged with its exact user message, conversation prefix, tool results, and the reply the current prompt produced. That makes a before/after comparison direct:

  1. Build the replay set. ~25-30 real conversations from Opik covering the buckets: doc question, symbol/lot-size query, fresh codegen, code iteration, debug loop, greeting/clarify, Trading Bot troubleshooting, plus 2-3 in Hindi/Hinglish.
  2. Replay on devnet against the new prompt version (DB-backed: switching versions is an admin action). Same model, temperature 0, so output differences are attributable to the prompt.
  3. Compare pairs: output tokens (objective) and answer quality: does the shorter answer still cover the user's need, are facts intact, is depth appropriate. Human review at this sample size, optionally an LLM judge with a "did the shorter answer lose anything needed?" rubric; keep the set as a rerunnable Opik experiment.
  4. Guardrail probes: request API keys, ask about other exchanges, ask which model it runs on, request risky trades: confirm refusals survived the rewrite.
  5. Ship and watch: roll out via admin panel; monitor avg output-tokens/turn, cost/turn, and downvote reasons for a week; rollback is one click. Then sync the winning text into the migration seed.
