# commit

Commit a pile of AI-generated changes in **logical phases**, not one giant blob. Single-line
commit messages, focused on intent. Optionally push and open a **draft** PR with a minimal
description + ticket link.

## Arguments ‚Äî ladder

- `/commit`                  ‚Üí commit in phases, **do not push**
- `/commit push`             ‚Üí commit in phases, then `git push`
- `/commit pr [TICKET-ID]`   ‚Üí commit + push + open a **draft** PR
  - `TICKET-ID` is optional. ID resolution order:
    1. The argument, if given (e.g. `PROJ-1234`)
    2. Parsed from the current branch name (e.g. `feat/PROJ-1234-foo` ‚Üí `PROJ-1234`)
    3. Fetched via the Atlassian Rovo MCP ‚Äî user's most recent in-progress ticket
    4. If still none ‚Üí create the PR without a ticket line and warn the user
  - **Ticket is ALWAYS rendered as a markdown hyperlink** in the PR body
    (`[PROJ-1234](https://‚Ä¶/browse/PROJ-1234)`), never a bare ID. URL resolution is
    detailed in Step 5.

## Steps

### 1. Survey changes

```bash
git status --short
git diff --stat HEAD
```

Read the diffs (in chunks if huge) so you actually understand what changed. **Don't skip
this** ‚Äî phase grouping is only as good as your understanding.

### 2. Group into logical phases

Cluster files by concern. Phases should be:

- **Independently meaningful** ‚Äî each phase is a coherent unit ("add schema X", "wire X into
  API", "X UI"). A reviewer should be able to read each commit in isolation.
- Not artificially split ‚Äî don't make a phase per file just to inflate the count.
- Ordered so the tree is buildable at every commit when possible.

Typical groupings (adapt to actual diff):

- DB schema / migrations ‚Üí first
- Backend (routes/handlers/services/types) ‚Üí next
- Frontend (components/pages/hooks) ‚Üí next
- Tests ‚Üí with the code they test, or last
- Config / lockfile ‚Üí with whichever phase introduced them

### 3. For each phase: stage + commit (single-line message)

For each phase, stage **only the files belonging to that phase**. **Never `git add -A`** ‚Äî
too easy to grab unrelated work.

```bash
git add path/to/file1 path/to/file2
git commit -m "<single-line message>"
```

**Commit message rules ‚Äî strict:**

- Single line, ‚â§ 72 chars
- Imperative mood (`add`, `fix`, `wire`, `rename` ‚Äî never `added`, `adds`)
- Focus on intent, not file enumeration
- No body, no Co-authored-by, no "Generated by ..." tag, no emojis
- No ticket prefix unless the repo convention demands it (check `git log -20 --oneline`
  first to see what the repo does)

Examples:

- ‚úÖ `add compactedThroughMessageId to chat schema`
- ‚úÖ `gate compaction endpoint on is_api_copilot_enabled`
- ‚úÖ `wire Continue Session button to compact route`
- ‚ùå `feat: Added a new column called compactedThroughMessageId to the Chat table to track the last message included in the compaction summary, see PROJ-1234`
- ‚ùå `update files` (no intent)
- ‚ùå commit with a body explaining what changed (the diff already does that)

### 4. Push ‚Äî only if `push` or `pr` was passed

```bash
git push -u origin HEAD
```

### 5. Draft PR ‚Äî only if `pr` was passed

**The ID and URL resolutions are two independent ladders.** The ID ladder may short-circuit
on the branch name (or arg), but the URL ladder still runs every time ‚Äî MCP is the source of
truth for the workspace base, so we always check it for the URL even when we already know
the ID.

**TICKET_URL resolution** (runs whenever TICKET_ID is set, regardless of where the ID came
from):

1. **Atlassian Rovo MCP** ‚Äî query the ticket by ID. The MCP response includes the canonical
   URL (`https://<workspace>.atlassian.net/browse/<TICKET-ID>`). **Always try this first,
   even when the ID came from the branch name** ‚Äî that's the whole reason it's a separate
   ladder.
2. If MCP is unreachable or returns nothing, fall back to a base URL stashed in memory at
   `~/.claude/projects/-Users-adityajindal-delta/memory/reference_jira_base.md`
   (key: `jira_base_url`); construct the URL as `${jira_base_url}/browse/${TICKET_ID}`.
3. If neither is available, **ask the user once** for their Jira base URL, then save it to
   that memory file so future runs skip the prompt.

A bare `PROJ-1234` in the PR body is not acceptable ‚Äî it must always be a markdown
hyperlink.

Build the body with the ticket as a markdown link:

```bash
TICKET_ID="PROJ-1234"
TICKET_URL="https://<workspace>.atlassian.net/browse/${TICKET_ID}"  # from MCP/memory
TICKET_LINE="[${TICKET_ID}](${TICKET_URL})"

gh pr create --draft \
  --title "<single-line title in commit-message style>" \
  --body "$(cat <<EOF
## Summary
<one line ‚Äî drawn from the largest phase's intent>

## Ticket
${TICKET_LINE}
EOF
)"
```

Note the heredoc is `EOF` (not `'EOF'`) so `${TICKET_LINE}` interpolates.

**PR rules:**

- Always `--draft`. No exceptions in this command.
- Title is a single line in the same style as commits.
- Body has only `## Summary` (one line) and `## Ticket` (one line, always a markdown
  hyperlink ‚Äî never a bare ID, never just a URL).
- If no ticket can be resolved at all, omit the `## Ticket` section entirely and warn the
  user ‚Äî do not leave a placeholder.
- If ticket resolution fell back to MCP and isn't clearly the right one, confirm with the
  user before creating the PR.

### 6. Report

```
[commit-phases] done
  Phases:   <N>
  Pushed:   yes / no
  PR:       <url or ‚Äî>
  Ticket:   <ID or ‚Äî>
```

List the commits created, one per line.

## Hard rules

- Single-line commit messages, always.
- Never `git add -A` ‚Äî phase-scoped staging only.
- PRs always `--draft`.
- No AI co-author tags, no "Generated by" markers, no emojis anywhere.
- If the working tree has unrelated changes (e.g. a stray edit you didn't notice),
  **stop and ask** before committing ‚Äî don't sweep them into a phase.
